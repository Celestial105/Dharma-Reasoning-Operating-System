| :--- | :--- | :--- | :--- |
| **🎮 顯示卡 (GPU)** | **0% (完全不使用)** | 所有複雜的 LLM 義理推理全部外包給 Google Cloud 雲端超級計算機，本地無需本地顯卡。 | **無任何顯卡要求** (內顯/核顯即可) |

## ⚡ 響應速度（首字流式輸出延遲）的三大物理鐵律

既然與本地電腦效能無關，那是什麼決定了 DROS 的「回應快慢」？主要取決於以下三個外部維度：

### 1. 🌐 網絡延遲 (Network Latency) —— **決定性佔比 70%**
* **原理**：DROS 本地後端需要透過 HTTPS 請求直連 Google AI Studio 的 API 伺服器。
* **速度體現**：您的電腦到 Google 伺服器的 **Ping 值（延遲）** 是最關鍵的。一個流暢、穩定的網絡代理（科學上網）節點，能讓首字噴射速度提升數倍。

### 2. 雲端大模型生成時間 (Google API Engine) —— **佔比 25%**
* **Gemini 2.5 Flash-Lite (快速伴學/路由分流)**：超高響應速度，首字噴射延遲通常在 **0.5 ~ 1 秒** 內。
* **Gemini 3.1 Pro (深度研經)**：由於進行高保真義理推理，首字響應會慢大約 **1 ~ 2 秒**，但給出的回答深度極高。

### 3. 是否觸發雲端 NotebookLM 檢索 (Fallback) —— **佔比 5%**
* **觸發機制**：當您提問的詞彙在本地 Wiki 中查無精確節點時，系統會自動透過 CLI 向 Google NotebookLM 雲端發送語義檢索。
* **延遲增量**：雲端向量 RAG 檢索會為本次對話帶來額外大約 **1 ~ 2 秒** 的網絡往返時間，隨後才會開始流式輸出。

## 🚀 閣主推介：三步極致加速法 (Optimization Tips)

如果您希望 DROS 的對話像打字機一樣飛速流暢，請嘗試以下三個優化方案：

### 1. ⚡ 優化網絡節點 (Choose the Best Network Route)
* 這是最有效的加速手段。請在您的網絡代理軟體中，選擇 **延遲最低 (Ping 值最小)、帶寬最穩定** 的節點（通常為美國或鄰近 Google 骨幹網的優質節點），這能瞬間將首字延遲縮短至 1 秒以內。

### 2. 📿 善用本地名相 Wiki (Build the Local Ontological Core)
* 儘量豐富您本地的 `core/` 或 `User_Pavilion/` 資料夾。當您提問的關鍵詞在本地有對應的名相 Markdown 檔案時，Trie 樹會直接命中，**0 延遲** 注入本地 Context，**完全免去調用雲端 NotebookLM 檢索的 2 秒等待時間**！

### 3. ⚖️ 選擇合適的契約模式 (Select the Right Contract)
* **Vajra (金剛模式)**：由於約束嚴格，要求回答極度簡煉且引經據典，生成字數少，**整體生成速度最快、消耗最省**。
* **Bodhisattva (菩薩模式)**：允許隨流聯想與現代詮釋，生成字數較多，適合在安靜的時間進行深入探討。
